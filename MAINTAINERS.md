# Project Maintainers

## Governance Model

**Status**: BDFN (Benevolent Dictator For Now)

This project currently follows a BDFN governance model, with plans to transition to a more distributed model as the community grows.

## Current Maintainers

### Lead Maintainer

- **Role**: Project Lead & BDFN
- **Responsibilities**:
  - Final decision authority on project direction
  - Release management
  - Security incident response
  - Community health and code of conduct enforcement
  - Strategic planning

### Core Maintainers

_Currently seeking core maintainers - see [Becoming a Maintainer](#becoming-a-maintainer)_

## Maintainer Responsibilities

### All Maintainers

1. **Code Review**: Review and merge pull requests
2. **Issue Triage**: Label, prioritize, and respond to issues
3. **Community Support**: Answer questions and provide guidance
4. **Code Quality**: Ensure adherence to project standards
5. **Documentation**: Maintain and improve project documentation
6. **Security**: Respond to security reports within SLA
7. **Testing**: Ensure adequate test coverage
8. **Releases**: Participate in release process

### Lead Maintainer Additional Responsibilities

1. **Governance**: Define and evolve project governance
2. **Conflict Resolution**: Final arbiter of disputes
3. **Roadmap**: Define project roadmap and priorities
4. **Community Growth**: Foster community development
5. **Legal**: Handle legal and licensing issues
6. **Funding**: Seek and manage project funding (if applicable)

## Decision-Making Process

### Routine Decisions

- **Minor Changes**: Single maintainer approval (any maintainer)
- **Features**: Two maintainer approvals
- **Breaking Changes**: All maintainer consensus + community notice

### Major Decisions

Major decisions require:

1. **Proposal**: Written proposal in GitHub Discussion
2. **Community Input**: 14-day comment period
3. **Maintainer Discussion**: Synchronous or asynchronous discussion
4. **Decision**: BDFN makes final decision considering community input
5. **Documentation**: Decision rationale documented

Major decisions include:
- Governance changes
- License changes
- Major architectural changes
- Deprecation of significant features
- Addition of new maintainers

### Deadlock Resolution

In case of maintainer disagreement:

1. **Discussion**: Extended discussion period (7 days)
2. **Community Input**: Seek community perspectives
3. **BDFN Decision**: Lead maintainer makes final decision
4. **Document Rationale**: Decision and reasoning documented

## Becoming a Maintainer

We welcome new maintainers who demonstrate:

### Criteria

1. **Sustained Contribution**: Consistent, quality contributions over 6+ months
2. **Technical Excellence**: Deep understanding of codebase
3. **Community Engagement**: Helpful, constructive engagement with community
4. **Code Review**: History of thoughtful code reviews
5. **Alignment**: Alignment with project values and goals
6. **Availability**: Commitment to ongoing maintenance responsibilities

### Process

1. **Nomination**: Self-nomination or nomination by existing maintainer
2. **Discussion**: Private discussion among current maintainers
3. **Trial Period**: 3-month trial period with limited permissions
4. **Evaluation**: Assessment of trial period performance
5. **Decision**: BDFN makes final decision
6. **Onboarding**: Technical and governance onboarding
7. **Announcement**: Public announcement of new maintainer

## Stepping Down

Maintainers may step down at any time:

1. **Notice**: Provide notice to other maintainers (no minimum required)
2. **Transition**: Transfer responsibilities and knowledge
3. **Access**: Repository access revoked
4. **Recognition**: Emeritus status and continued recognition
5. **Gratitude**: Public thanks for service

### Emeritus Maintainers

Former maintainers who stepped down in good standing may receive:

- Emeritus maintainer title
- Recognition in project documentation
- Advisory role (optional)
- Invitation to special discussions
- Path to return as active maintainer

## Maintainer Expectations

### Time Commitment

- **Minimum**: 4 hours per week
- **Typical**: 8-16 hours per week
- **Variable**: Workload varies with release cycles

### Availability

- **Response Time**: Respond to mentions within 48 hours
- **Code Review**: Review PRs within 7 days
- **Security**: Respond to security reports within 24 hours
- **Absences**: Announce extended absences (>2 weeks) in advance

### Conduct

All maintainers must:

- Follow the [Code of Conduct](CODE_OF_CONDUCT.md)
- Model inclusive and welcoming behavior
- Mentor and support contributors
- Provide constructive feedback
- Acknowledge their mistakes and apologize when appropriate
- Accept feedback gracefully

## Maintainer Removal

A maintainer may be removed for:

1. **Inactivity**: No participation for 6+ months without notice
2. **Conduct Violations**: Serious or repeated Code of Conduct violations
3. **Breach of Trust**: Security violations, abuse of access
4. **Conflict of Interest**: Unresolved conflicts of interest

### Removal Process

1. **Private Discussion**: Issue raised with other maintainers
2. **Conversation**: Attempt to resolve issue directly
3. **Formal Review**: If unresolved, formal review by BDFN
4. **Decision**: BDFN makes final decision
5. **Communication**: Private notification to affected maintainer
6. **Access Revocation**: Repository and system access removed
7. **Optional Announcement**: Public announcement (decision by BDFN)

## Governance Evolution

### Future Plans

As the project and community grow, we plan to evolve governance:

1. **Expand Core Team**: Add 3-5 core maintainers
2. **Technical Steering Committee**: Create TSC for technical decisions
3. **Working Groups**: Form specialized working groups
4. **Consensus Model**: Transition to consensus-based decision making
5. **Community Representation**: Add community representatives

### Triggering Evolution

Governance evolution will be considered when:

- 5+ active regular contributors
- 50+ pull requests per month
- Significant community size (1000+ users)
- Consistent maintainer workload >20 hours/week
